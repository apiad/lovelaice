# Lovelaice v1 — design

A sovereign, local-first coding agent for the terminal. v1 polishes and
extends the v0.7 scaffold (`Config`/lingo/ReAct/yolo) with a full-screen
TUI, reasoning-model passthrough, locked-down workspace-grounding
semantics, an MCP-compatible tool surface, and a clean one-shot mode.

## Goals

- Single ReAct loop over a small built-in tool set, yolo by default.
- Two modes: **interactive** (full-screen Textual TUI) and **one-shot**
  (`lovelaice <prompt>` → streams Rich Markdown to stdout, exits).
- Workspace is rooted at the nearest ancestor `.lovelaice.py`. The
  binary `chdir`s there before doing anything else.
- Reasoning-model thinking rendered in the transcript whenever the
  model emits it. OpenRouter only.
- MCP-compatible: stdio servers configured in `.lovelaice.py`, their
  tools register alongside the built-ins.
- Custom tools and commands via `@config.tool` / `@config.command` in
  `.lovelaice.py`.

## Non-goals

- Approval prompts (yolo).
- Sandboxing.
- `.lovelaice.py` stacking. Exactly one config grounds the workspace.
- Built-in web search (custom-tool territory).
- `think` tool (deferred to v1.x — re-evaluate after we have signal
  from non-reasoning model usage).
- VS Code / IDE integration.
- Session persistence and resume.
- Context compaction / summarization.
- Multi-agent.

## §1 — Workspace grounding

On startup, walk upward from `os.getcwd()` looking for `.lovelaice.py`.
First match wins. If none → exit with `No .lovelaice.py found. Run
'lovelaice --init' first.` (exit code 1).

When found, `os.chdir()` to that file's directory **before** loading
the config. Tools, `bash`, the agent's view of `cwd` — all see the
workspace root. The original launch directory is dropped on the
floor: not stashed, not exposed, not in the system prompt.

The system prompt advertises the workspace root as `cwd`. There is
no other "workspace" abstraction.

`find_config_file()` already exists in v0.7 with the right walking
behavior; v1 only adds the `chdir` and inlines the not-found error
into the CLI entry.

## §2 — TUI (interactive mode)

**Framework.** [Textual](https://textual.textualize.io/).

**Layout.** Single-pane, top to bottom:

- **Header bar.** App name · current model alias · workspace root
  (left-truncated when long).
- **Transcript** (middle, scrolls). Every block in conversational
  order: user messages, agent replies (streaming Markdown), thinking
  blocks (collapsible, dim/italic), tool calls (one-line summary +
  collapsible result body), tool errors (red).
- **Input box** (bottom, multiline). Grows up to 5 lines, then
  scrolls internally. `Enter` submits, `Shift+Enter` / `Ctrl+J` newline.
- **Footer / status line.** Keybindings + transient status
  (`thinking…`, `running bash…`, token count, last error).

**Streaming.** Tokens stream into whichever block is active. Textual's
`Markdown` widget handles incremental re-render without flicker.

**Interaction.**

| Key | Action |
|---|---|
| `Enter` | Submit. |
| `Shift+Enter`, `Ctrl+J` | Newline in input. |
| `Ctrl+C` | Cancel current turn (see §6). Second within ~1s → quit. |
| `Ctrl+D` | Quit. |
| `↑` / `↓` (empty input) | Recall previous user messages (in-session, not persisted). |
| `PageUp` / `PageDown` | Scroll transcript. |
| Mouse | Scroll. Click to expand/collapse a block. |

**Collapsibility.** Thinking blocks and tool result bodies render
collapsed by default once the block finishes streaming. A one-line
summary stays visible. Click or focus + `Enter` to expand. Reduces
visual noise on long sessions.

**Theme.** Dark only. No theme config.

**Out of scope for v1.** Side panes. File-tree pane. Diff viewer.
Tabs. Multi-session. Fuzzy file picker. Image rendering.

## §3 — One-shot mode

`lovelaice <prompt>` skips Textual entirely. A Rich `Live` panel
streams the agent's working transcript to stdout, the agent runs
to completion, lovelaice exits.

**Render order.**

1. Thinking blocks (dim/italic Rich panels) — only if the model
   emits reasoning.
2. Tool calls (one line each, header-only summary): `Read X (n
   lines)`, `Wrote Y (n lines)`, `Ran bash`. Result bodies suppressed.
3. Final agent reply, streamed live into a `Lovelaice` Rich panel.

One-shot mode renders linearly with no collapsing: thinking and tool
blocks stay expanded as written. The collapsing behavior is a TUI
affordance only.

**Flags.**

- `--verbose` — also render full tool result bodies (debugging).
- `--model <alias>` — override the default model alias for this turn.

**Exit codes.**

- `0` — success (agent emitted a final reply).
- `1` — loop hit `max_steps` without a final reply.
- `2` — tool error or LLM transport error that terminated the loop.

**Pipe / redirect detection.** When `sys.stdout.isatty() == False`,
drop Rich entirely and emit only the final agent reply as plain text
to stdout. Thinking, tool calls, panels — all suppressed. This makes
`lovelaice "summarize X" | pbcopy` and `$(lovelaice ...)` substitution
work cleanly. `--verbose` overrides this and emits the full transcript
to stderr while keeping the reply on stdout.

## §4 — Thinking mode (OpenRouter only)

**Configuration.** Each entry in `MODELS` may carry a `thinking=` knob:

```python
"deep": {
    "model": "anthropic/claude-sonnet-4-5",
    "api_key": os.getenv("OPENROUTER_API_KEY"),
    "base_url": "https://openrouter.ai/api/v1",
    "thinking": "high",   # "low" | "medium" | "high" | int (token budget) | None
},
```

- `thinking=None` (default) — no reasoning request; behaves like a
  normal chat-completions call.
- `thinking="low"|"medium"|"high"` — translated to OpenRouter's
  unified `reasoning: { effort: ... }` request body.
- `thinking=<int>` — explicit budget, sent as
  `reasoning: { max_tokens: <int> }`.

**Streaming.** Lovelaice consumes `delta.reasoning` chunks from the
SSE stream and routes them to the thinking block. `delta.content`
chunks continue to feed the reply block. If a model is configured
with `thinking=` but the response carries no reasoning content, no
thinking block is rendered (silent no-op).

**Implementation.** Lingo's `LLM` does not natively support
reasoning. Lovelaice owns the integration in a `lovelaice/thinking.py`
module:

- Inject `reasoning: { ... }` into the request body when the model
  config carries `thinking=`.
- Parse `delta.reasoning` from streamed chunks and route to a separate
  callback (the thinking-token sink).
- Leave `delta.content` handling untouched.

The hook point into lingo's streaming layer is an open implementation
question (see §10). The realistic options are:

1. Wrap lingo's `LLM` in a `ThinkingLLM` adapter that re-implements
   the streaming call against the OpenRouter SSE protocol when
   `thinking` is set, falling through to lingo's default otherwise.
2. Subclass `lingo.LLM` if it exposes a stream-chunk hook.

The decision lands at writing-plans time after a brief read of
lingo's LLM source.

**Non-reasoning models.** No fake-thinking simulation. No
`<thinking>` prompt-engineering. Models that cannot emit reasoning
just don't render thinking blocks. The optional `think` tool is a
v1.x conversation, not v1.

**Other base URLs.** Lovelaice still accepts arbitrary OpenAI-
compatible `base_url`s for chat/tools (LM Studio, vLLM, Ollama,
direct provider APIs). The `thinking=` knob is silently ignored on
non-OpenRouter base URLs — translating to every provider's bespoke
reasoning protocol is explicitly out of scope.

## §5 — Tool surface

**Built-ins** (registered by default in the `--init` template).

| Tool | Signature | Notes |
|---|---|---|
| `bash` | `bash(command: str) -> str` | Runs in workspace root. Captures stdout+stderr (combined into a single string in invocation order). Timeout configurable via `Config(bash_timeout=...)`, default 120s. Timeout → `tool failed: bash timed out after Ns`. |
| `read` | `read(path: str) -> str` | Same as v0.7. |
| `write` | `write(path: str, content: str) -> str` | Same as v0.7. |
| `edit` | `edit(path: str, old: str, new: str) -> str` | Same as v0.7 — fails if `old` is absent or non-unique. |
| `list` | `list(path: str = ".") -> list[str]` | Renamed from v0.7's `list_dir`. Defined as `list_` in `tools/files.py`, registered with display name `list` to avoid shadowing the builtin in user code. |
| `glob` | `glob(pattern: str) -> list[str]` | Recursive `**/*` patterns supported, runs from workspace root, gitignore-respecting via `pathspec` against `.gitignore`. |
| `grep` | `grep(pattern: str, path: str = ".") -> str` | Literal-or-regex search, returns `path:line:text` lines. Caps at 200 hits. |
| `fetch` | `fetch(url: str) -> str` | HTTP GET, follows redirects. `text/html` → markdown via `markdownify` (with a readability pre-pass). Other `text/*` returned as-is. ~50 KB cap. |

**Custom tools.** `@config.tool` on any async function. Function
docstring → tool description; signature → schema (typed parameters
with defaults). Raised exceptions are caught at the loop boundary
and surfaced as a `tool failed` observation; the loop continues.

**MCP.**

`Config` gains an `mcp=` arg accepting a list of MCP server specs:

```python
config = Config(
    models=MODELS,
    prompt=PROMPT,
    mcp=[
        {"name": "filesystem", "command": "npx",
         "args": ["@modelcontextprotocol/server-filesystem", "."]},
        {"name": "github", "command": "uvx",
         "args": ["mcp-server-github"],
         "env": {"GITHUB_TOKEN": os.getenv("GITHUB_TOKEN")}},
    ],
)
```

At `Config.build()` time:

1. Spawn each server as a subprocess (stdio transport only in v1;
   HTTP/SSE transport deferred).
2. Perform the MCP handshake.
3. Fetch each server's tool list.
4. Register each tool on the agent with a `mcp:<server>:<tool>`
   display name.

Servers are torn down on agent exit. Handshake failure → log + skip
that server, continue. Mid-session crash → red banner, that server's
tools become unavailable for the rest of the session, others keep
working.

Dependency: the official `mcp` Python SDK (`pip install mcp`).

## §6 — Slash commands and lifecycle

**Slash commands** (TUI only, handled in the input layer before any
LLM call):

| Command | Behavior |
|---|---|
| `/help` | Modal listing slash commands + keybindings. |
| `/model` | List aliases with the current one starred. |
| `/model <alias>` | Switch active model for the next turn; update header. |
| `/clear` | Wipe the in-memory `lingo.Context`. Transcript pane keeps the scrollback with a divider rendered at the boundary. |
| `/cost` | Cumulative input/output tokens since launch (and reasoning tokens if any). No dollar conversion. |
| `/cwd` | Print the workspace root. |
| `/exit`, `/quit` | Same as `Ctrl+D`. |

**User-defined commands** (`@config.command`) are *not* exposed as
slash commands. They remain agent-side workflows registered as lingo
skills (the v0.7 pattern), invoked via natural language. Slash
commands are exclusively user→TUI; agent-side commands are
exclusively user→agent. Mixing the two confuses the mental model;
keep them separate.

**Cancellation.**

- `Ctrl+C` mid-turn → propagate `asyncio.CancelledError` up through
  the ReAct loop. In-flight tool calls are cancelled (subprocess for
  `bash` → SIGTERM after 1s grace, then SIGKILL). The transcript
  keeps everything emitted up to that point with a red `↳ cancelled
  by user` line appended. Context retains the partial state — next
  user message continues coherently.
- Two `Ctrl+C` within ~1s → quit the app.

**Sessions.** In-memory only for v1. No on-disk session file, no
resume, no history search. Closing the TUI loses the conversation.
The `↑` / `↓` input recall is per-process. Defer persistence to v1.x.

**Compaction.** Deferred. v1 lets context grow until the model errors.
The agent-emitted environment block in `core.py` is regenerated each
turn (already true in v0.7). Real usage will tell us what shape
compaction needs.

**Errors.**

- LLM transport error → red banner: `LLM error: <type>: <message>`.
  Loop terminates. User can type the next message; we re-try then.
- Tool exception → red collapsible block: `tool <name> failed:
  <message>`. Result fed back to the agent as an observation; the
  loop continues.
- MCP server crash → handled as above (§5 MCP).

## §7 — `.lovelaice.py` template

What `lovelaice --init` writes (after prompting for default model and
API key, which substitute into `MODELS["default"]`):

```python
import os
from lovelaice import Config

# --- Models ---------------------------------------------------------------
# All models go through OpenRouter. The first entry is the default;
# pick another at runtime with `--model <alias>` (one-shot) or `/model`
# (TUI). Add `thinking="low"|"medium"|"high"` to opt into reasoning.
MODELS = {
    "default": {
        "model": "google/gemini-2.5-flash",
        "api_key": os.getenv("OPENROUTER_API_KEY"),
        "base_url": "https://openrouter.ai/api/v1",
    },
    # Example: a reasoning-enabled alias
    # "deep": {
    #     "model": "anthropic/claude-sonnet-4-5",
    #     "api_key": os.getenv("OPENROUTER_API_KEY"),
    #     "base_url": "https://openrouter.ai/api/v1",
    #     "thinking": "high",
    # },
}

# --- System prompt --------------------------------------------------------
PROMPT = """
You are Lovelaice, a sovereign coding agent that runs in the user's terminal.
Be concise and act decisively. When a task is ambiguous, ask one focused
question instead of guessing. Prefer surgical edits over full rewrites.
""".strip()

# --- Build the config -----------------------------------------------------
config = Config(
    models=MODELS,
    prompt=PROMPT,
    # mcp=[{"name": "...", "command": "...", "args": [...]}],
)

# --- Default tools --------------------------------------------------------
from lovelaice.tools import bash, read, write, edit, list_, glob, grep, fetch

config.tool(bash)
config.tool(read)
config.tool(write)
config.tool(edit)
config.tool(list_, name="list")
config.tool(glob)
config.tool(grep)
config.tool(fetch)

# --- Default command: ReAct loop ------------------------------------------
from lovelaice.commands import react
config.command(react)

# --- Custom tools and commands --------------------------------------------
# @config.tool
# async def search_notes(query: str) -> str:
#     """Search the user's notes."""
#     ...
```

## §8 — Module layout

```
src/lovelaice/
├── __init__.py          # exports: Config
├── cli.py               # typer app, mode dispatch (one-shot vs TUI)
├── config.py            # Config, find_config_file, load_agent_from_config
├── core.py              # Lovelaice (lingo subclass) + thinking-aware LLM wrapper
├── thinking.py          # OpenRouter `reasoning` request/response handling
├── template.py          # template rendered by `lovelaice --init`
├── tui/
│   ├── __init__.py
│   ├── app.py           # Textual App + screens
│   ├── transcript.py    # transcript widget (renders blocks)
│   └── blocks.py        # message / thinking / tool block widgets
├── oneshot.py           # Rich-based one-shot renderer + isatty pipe mode
├── mcp.py               # MCP server lifecycle + tool registration
├── commands/
│   ├── __init__.py
│   └── react.py         # default ReAct command (mostly v0.7 as-is)
└── tools/
    ├── __init__.py      # exports built-ins
    ├── bash.py          # bash, with timeout
    ├── files.py         # read, write, edit, list_
    ├── search.py        # glob, grep
    └── web.py           # fetch
```

Surviving v0.7 essentially as-is: `commands/react.py` (loop logic is
good), `tools/bash.py` (gains a timeout), `tools/files.py` (rename
`list_dir` → `list_`).

Changes to existing files:

- `cli.py` — split mode dispatch into `tui/app.py` (interactive) vs
  `oneshot.py` (non-interactive); add the `chdir` step; add the
  not-found error path; move `_LiveChat` into `oneshot.py`.
- `config.py` — `Config(__init__)` accepts `mcp=` and
  `bash_timeout=`; `Config.tool()` accepts an optional `name=`
  override; `build()` initializes MCP servers and registers their
  tools; `find_config_file` unchanged.
- `core.py` — env-block generation also surfaces MCP-loaded tools;
  `LLM` construction routes through `thinking.py` when the model
  config carries `thinking=`.
- `template.py` — replaced wholesale per §7.

New files: `thinking.py`, `tui/*`, `oneshot.py`, `mcp.py`,
`tools/search.py`, `tools/web.py`.

## §9 — Open implementation questions

To resolve at writing-plans time, not blocking the spec:

1. **Lingo streaming hook.** Does `lingo.LLM` expose a per-chunk
   callback that lets us peek at `delta.reasoning`, or do we need
   to wrap the streaming HTTP call ourselves? The shape of
   `thinking.py` depends on the answer (adapter vs subclass).
2. **Lingo's tool-call protocol vs MCP tools.** MCP tools have a
   richer schema (typed parameters, descriptions, sometimes JSON
   Schema constraints). Confirm lingo's tool registration accepts
   this or wrap on the lovelaice side.
3. **Textual + asyncio + lingo.** Lingo runs the loop on `asyncio`
   already; Textual is also asyncio-native. Confirm there's no event
   loop conflict — the expected shape is "Textual's app runs the
   loop, agent calls schedule onto it." Brief smoke test before
   committing to the layout.
4. **Token counters for `/cost`.** Whether OpenRouter returns usage
   on streaming responses (it does, in the final `usage` chunk),
   and whether lingo surfaces it. Likely we accumulate the final-
   chunk usage ourselves.

These are verification tasks for the implementation plan, not spec
ambiguities.

## §10 — Out-of-scope and v1.x candidates

- `think` tool for non-reasoning models.
- Session persistence and resume.
- Context compaction / summarization.
- HTTP/SSE MCP transport.
- Multi-provider thinking translation (Anthropic native, OpenAI
  o-series native).
- Slash command for fuzzy file insert (`@path/to/file`).
- `--json` output mode for one-shot.
- Light theme.
- Dollar-cost accounting in `/cost`.
